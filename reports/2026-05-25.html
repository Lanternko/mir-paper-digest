Towards Effective Negation Modeling in Joint Audio-Text Models for Music

讓 CLAP 類 joint embedding 真的「懂否定」：用文字增強 + dissimilarity loss 把「否定後的 caption」從原 caption 拉開，降低檢索/生成被語義陷阱誤導。

Tags CLAP / joint audio-text LP-MusicCaps-MSD evaluation pitfalls data augmentation
新穎性

不是新模型，而是補「否定語義」這個常見盲點；對「用 CLAP 做 data filtering / prompt selection / auto-eval」特別重要。

為何入選

你的主線常會用到 text-audio similarity（或其衍生）做篩選與評估；否定處理若不可靠，很容易讓資料清理與評估一起偏。

來源

摘要

作者指出 joint audio-text 模型在否定描述（如「沒有鼓聲」）上容易失效。做法是用文字增強產生「否定版 caption」，再配合 dissimilarity-based 對比式損失，讓否定後的語意在 joint space 明確遠離原 caption，並在音樂資料上驗證。

可行性（你能怎麼做）

  • 先用你現有的 CLAP/embedding pipeline 做「否定 prompt 對照測試」（檢索/重排序）。
  • 把否定樣式納入資料過濾：避免把「沒有 X」錯當「有 X」。
  • 若你自訓 CLAP 或 reward model：把 negation augmentation 當成低成本增強。

主要貢獻（快速抓重點）

  • 在音樂資料上系統化呈現「否定語義」的失效案例。
  • 提供可重用的 negation text augmentation + dissimilarity loss 設計。
  • 用 LP-MusicCaps-MSD（Million Song Dataset + pseudo captions）場景連到實務。

和 MeanAudio 的關係

如果你用 CLAP 類分數做：caption quality filtering、hard negative、LLM 產生 captions 的篩選、或自動化 eval（text alignment），這篇能提醒「否定/反例」會把分數弄歪，並給一個可落地修補方向。

下一步實驗

  • 挑 30 個你常用的音樂 prompt，為每個 prompt 造 1 個否定版本，測 CLAP 排名差異與人耳一致性。
  • 把 negation 變成 data filter 的「紅旗」規則：否定 prompt 命中高分 → 需要人工審核。

限制與注意事項

否定只是語義可靠性的其中一塊；仍要注意 domain shift（speech/music）、以及 pseudo captions 的噪音會不會把「否定增強」學偏。不要只用單一 embedding 指標做最終結論。

STARS: A Unified Framework for Singing Transcription, Alignment, and Refined Style Annotation

把唱歌資料的「轉寫、對齊、技巧/風格標註」做成一條統一 pipeline：更像你真的要做 karaoke / SVS 資料工程時會需要的那種 end-to-end 工具。

Tags singing transcription phoneme/audio alignment style / technique labels dataset building
新穎性

亮點在「把多個分散任務整合」：不只是單點 SOTA，而是讓標註管線變得可規模化。

為何入選

karaoke-jp 很容易被資料標註卡住（lyrics/phoneme/音高/節拍/時間對齊）；這篇提供一個可參考的整體設計。

來源

摘要

STARS 同時處理唱歌轉寫、音素/詞級對齊、以及更細的風格/技巧標註。透過階層式的聲學特徵處理與非自回歸的在地編碼器，支援多層級結構（frame/word/phoneme/note/sentence），並展示在多個面向的改進。

可行性（你能怎麼用）

  • 拿它當「自動標註器」原型：先做對齊/轉寫，再把結果餵回你的 karaoke 渲染。
  • 把它當成你要蒐集/清理唱歌資料時的參考設計（多層級 supervision）。
  • 若你在意日文：重點是 phoneme 對齊設計能否接到 kana/ruby/mora。

主要貢獻（快速抓重點）

  • 把轉寫 + 對齊 + 風格標註整合成單一框架。
  • 提供更可規模化的自動唱歌標註路線，服務 SVS / karaoke 資料需求。

和 karaoke-jp 的關係

你可以用這條思路把「lyrics → phoneme → timing」串起來，進一步做 mora-to-note / LRC timecode / 逐字高亮。它也提醒：資料管線比單點模型更可能成為瓶頸。

下一步實驗

  • 挑 5 首日文歌：用現成 ASR/lyrics + 強制對齊做 baseline，對比 STARS 類方法在對齊穩定性。
  • 以「karaoke 可讀性」當最終指標：逐字高亮是否對得上、錯誤是否集中在連音/促音/長音。

限制與注意事項

唱歌對齊容易受到伴奏、混響與歌手咬字影響；如果模型訓練資料偏某些語言/風格，日文歌曲可能需要 domain adaptation。建議把錯誤分佈做可視化（對齊偏移、漏字、重複）。

今天的 take-away

1) 對 MeanAudio：你如果靠 CLAP / joint embedding 做資料篩選與評估，務必把「否定/反例」納入 sanity check，否則很容易越篩越偏。
